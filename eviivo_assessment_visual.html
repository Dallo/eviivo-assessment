A conversation is the container grouping all communication between a property and guest for a specific reservation. Each reservation auto-creates an empty conversation, identified by a conversation_id and linked to the booking via conversation_reference. It always has exactly two participants: property and guest.

A message is an individual piece of content within a conversation — with its own message_id, timestamp, sender, and content. From API v1.2, messages also carry a message_type (free_text, email, automatically_sent_template, self_service_event, or request_to_book_event).

Why this matters for eviivo

The separation enables a clean, reservation-scoped inbox. Tags operate at different levels — read applies to individual messages, while no_reply_needed applies to the whole conversation — reinforcing why they are architecturally distinct.

1b) POST Example and Why

POST /messaging/properties/{property_id}/conversations/{conversation_id} { "message": { "content": "Welcome! We look forward to hosting you.", "attachment_ids": [] } }

POST is required because sending a message creates a new resource. Each call produces a unique message_id — this is not idempotent (calling twice creates two messages).

1c) Product Applications of Message Tags

Booking.com currently supports two property-facing tags: message_read and no_reply_needed. eviivo could extend the concept within its own product:

🔴 Priority / Urgency
Flag conversations as urgent, normal, or low. AI Concierge could auto-suggest priority based on content (e.g., allergy mentions → urgent).
🏷️ Topic / Category
Tags like "check-in", "parking", "payment", "complaint", "upsell" — enabling reporting on what guests most commonly ask. AI-powered detection extends this beyond Booking.com's structured requests.
⏰ Follow-up Required
Beyond "no reply needed" — a "follow-up required" tag with optional date/time reminder prevents messages falling through the cracks.
😊 Sentiment
AI auto-tags sentiment (positive/neutral/negative). Properties spot unhappy guests before they leave negative reviews — enabling proactive service recovery.
💰 Revenue Opportunity
When a guest mentions upgrades, extended stays, or additional services, an "upsell" tag routes these to a specific workflow — directly tying messaging to revenue generation via eviivo's Performance Manager.

eviivo opportunity: AI Concierge auto-responds using property data already in eviivo Suite. A guest requests parking info at 11pm → Concierge retrieves parking details from Property Manager → sends instant reply. No human intervention needed.

Enables property-guest messaging before booking confirmation. Crucial for vacation rentals where guests ask questions before committing. eviivo could surface these with a distinct visual indicator in Guest Manager and track how response times correlate with booking conversion rates.

1e) Deprecation Notice — Migration Work

"We are soon going to deprecate retrieving latest messages by using the GET /messages/latest endpoint and confirming their retrieval using the PUT /messages endpoint. Use the Connectivity Notification Service to receive notifications of newly created messages."

This is a shift from polling to push-based notifications:

Current (Being Deprecated)

Polling model

eviivo calls GET /messages/latest periodically → confirms via PUT /messages → repeat. Wasteful when no new messages exist. Risk of latency between polls.

New (CNS Push Model)

Event-driven model

Booking.com pushes a notification to eviivo's endpoint when a new message arrives → eviivo fetches message details. Instant. Efficient. No wasted calls.

How I would scope and guide this work with a junior developer:

1

Webhook endpoint setup

Expose an HTTPS endpoint for Booking.com's CNS to POST notifications to. Must be highly available, authenticated, and return 200 OK quickly. Configure in the Provider Portal. Walk the developer through CNS payload structure.

2

Notification-triggered message retrieval

When CNS notification arrives, call GET /conversations endpoints to fetch actual message content. The notification signals something happened; we still fetch the details. "Notification + fetch" pattern.

3

Run both systems in parallel

Keep polling active alongside the new push model during transition. Compare messages received by both using message_id as deduplication key. Only deprecate polling once push achieves 100% coverage.

4

Remove deprecated code

Once validated, remove the polling cron job and PUT /messages confirmation calls. Simplifies codebase and reduces API call volume.

5

Error handling and resilience

Handle endpoint downtime (retry + fallback to GET /messages/search), duplicate notifications (idempotent processing via message_id), and out-of-order delivery.

6

Testing in sandbox

Use Booking.com's test property environment. Create test reservations, send messages as both property and guest, verify notifications arrive, confirm messages display in Guest Manager.

7

Monitoring and alerting

Dashboard tracking notification receipt rate vs expected message volume. Alert on delivery drops or latency spikes.

Guiding the Junior Developer
  • Walk through the full Messaging API documentation structure before any code
  • Explain token-based authentication and Provider Portal configuration
  • Pair-program the first webhook endpoint, especially security aspects
  • Create clear acceptance criteria per workstream
  • Set up sandbox test property and walk through a full flow together first

2a) Advising Andre

Andre's query has a logical error in the WHERE clause:

Andre's Query (Broken)
WHERE LastName = 'White' AND LastName = 'Whyte' -- Returns 0 rows. A field -- can't equal two values -- simultaneously.
Corrected Query
WHERE LastName = 'White' OR LastName = 'Whyte' -- Returns rows matching -- either spelling.

The AND operator requires both conditions to be true for the same row — logically impossible for a single field. OR returns rows matching either value.

Alternative approach using pattern matching if Andre is unsure of all possible spellings:

WHERE LastName LIKE 'Wh%te' -- Matches White, Whyte, and other variations

2c) Average Books Borrowed per Customer

What we need to know first: The Customer table alone doesn't contain borrowing data. We need a Loans table (likely: LoanId, CustomerId FK, BookId FK, BorrowDate, ReturnDate).

SELECT AVG(BookCount) AS AverageBooksPerCustomer FROM ( SELECT c.CustomerId, COUNT(l.LoanId) AS BookCount FROM Customer c LEFT JOIN Loans l ON c.CustomerId = l.CustomerId GROUP BY c.CustomerId ) AS CustomerBorrowCounts

Key decisions in this query:

  • LEFT JOIN (not INNER JOIN) — includes customers who never borrowed a book, counted as 0. Critical for an accurate average.
  • Subquery approach — first counts per customer, then averages. Transparent and easy to extend.
